# i1-2: specialized routes still stamped Selected.* onto deserialized PSObjects

**Severity**: MEDIUM — persistent TypeNames mutation of caller objects for
the deserialized/remoting workloads ptk targets (EXO/AD implicit remoting,
Import-Clixml); no output corruption.
**Status**: Verified
**Branch**: master (direct, per the recorded codex-loop precedent)
**Commit**: (filled in at commit)

## Evidence
`src/PwshTokenCompressor.psm1` as of 86f990e: the generic path was fixed,
but the specialized compressors still piped caller-owned rows through
`Select-Object -First` — fs dirs/files, MatchInfo group rows, process
rows, service rows. The 86f990e reasoning ("raw .NET objects get
transient wrappers") missed that DESERIALIZED objects are persistently
PSObject-wrapped and match the specialized type guards (`*System.IO.
FileInfo*` matches `Deserialized.System.IO.FileInfo`). Codex probe: a
stored `Deserialized.System.IO.FileInfo` came back as
`Selected.System.Management.Automation.PSCustomObject`.

## Approach
New `Select-PtcFirst` helper (index-based head slice, empty for
non-positive counts) replaces every `Select-Object -First` over object
rows in the shaping code: fs dirs, fs files, MatchInfo groups and group
rows, process rows, service rows.

## Files changed
- `src/PwshTokenCompressor.psm1` — Select-PtcFirst; five call sites.

## Guard proof
- `tests/PwshTokenCompressor.Tests.ps1::does not mutate deserialized
  PSObjects routed to specialized compressors` — one Deserialized.* probe
  per route (fs, MatchInfo, process, service); each must keep its
  TypeNames[0] byte-exact after compression. Proven failing with the fs
  site reverted and with the MatchInfo site reverted, independently;
  passes with the fix.
