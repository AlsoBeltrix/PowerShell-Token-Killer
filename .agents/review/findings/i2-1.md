# i2-1: dev-install's hook-consent gate was a raw text match

**Severity**: MEDIUM — `ptk-hook.ps1` appearing anywhere in
`~/.claude/settings.json` (permissions lists, other hook events) would
count as prior hook consent and a `-Hook`-less install would install a
blocking hook.
**Status**: Verified
**Branch**: master (direct)
**Commit**: `665d99a`

## Evidence / failure
`scripts/dev-install.ps1` (as of the slice-2 commit): `(Get-Content
$settings -Raw) -like '*ptk-hook.ps1*'` gated the refresh; ptk_init then
installs a NEW entry when none actually exists.

## Approach
`Test-PtkHookEntryPresent` parses the settings JSON and matches the marker
only inside `hooks.PreToolUse[].hooks[].command` (mirrors ptk_init's
tested entry shape). Used by the install refresh gate AND the uninstall
path's identical raw match.

## Guard proof
Not automated — dev-install has no test harness (recorded precedent); the
helper mirrors ptk_init's Pester-covered entry-shape logic. Codex re-grade
covers it.
