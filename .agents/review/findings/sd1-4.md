# sd1-4: session-shadowed `set` still false-positives (contests a frozen decision)

**Severity**: LOW — requires the user to alias-shadow `set` over the built-in
`set → Set-Variable` alias; narrow input class, but once wired (slice 2) valid
session PowerShell would be refused.
**Status**: CONTESTED — challenges the frozen slice-0 decision; awaiting owner adjudication
**Branch**: n/a (no fix authorized)

## Evidence
Reported by codex at re-grade round 1 (head `acb0f39`): the `set` branch
(psm1:685) is unconditional — sd1-1's resolution guard deliberately exempts
`set` because it resolves (alias) in every pwsh by design, per the frozen
slice-0 decision keying it to argument shape instead.

**Master verified live at head:** with `Set-Alias set __shadowset` (function
accepting `[switch]$e`), `set -e` executes (`shadow-ok`) yet
`Get-PtcShellDialectFinding -Script 'set -e'` returns
`bash 'set' shell options (set -e/-u/-x/-o pipefail)`.

## Predicted observable failure
A session that legitimately overrides the `set` alias gets its working
`set -e` refused once the detector is wired to the refusal path.

## Why contested rather than admitted
The false positive is real, but the exemption was an explicit, recorded
slice-0 trade-off (argument-shape keying beats resolution for `set`, since
`set` ALWAYS resolves and shape is the only discriminator). Honoring a
session resolution check here means distinguishing the default
`set → Set-Variable` alias from user overrides — doable (compare resolved
target), but it reopens the frozen decision, so it is the owner's call, not
the loop's.
