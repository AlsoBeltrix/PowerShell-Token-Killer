# sd3-3: Job-poll recovery note fires on marker text the job itself printed

**Severity**: LOW — a spurious note prompts one unnecessary log retrieval;
no wrong execution.
**Status**: In progress
**Branch**: master (direct, per recorded precedent)
**Commit**: (filled in after commit)

## Evidence
`server/PtkMcpServer/Tools/JobTool.cs` (sd3-2 fix) infers elision from an
origin-blind substring search of the shaped poll, while the module
preserves under-limit passthrough text unchanged — so a job whose OWN
output contains the marker wording (a background grep over this repo's
source, a job cat-ing previously elided output) receives the recovery
note despite complete output.

## Predicted observable failure
An under-limit poll carrying job-printed marker text appends "raw=true
does not apply here - the complete raw log is <path>", prompting a
pointless read of a log that contains nothing beyond what the poll
already showed.

## What
The note must key on "shaping elided THIS poll", not "the poll contains
elision-marker-shaped text".

## Approach
Two gates replace the substring: (1) an anchored whole-line regex — a
genuine marker is always its own line, so quoted/prefixed source-code
copies no longer match; (2) shaping must have actually shortened the text
(trailing whitespace normalized) — under-limit passthrough is unchanged,
so even an exact marker-line copy cannot qualify. Recorded residual: rtk
log-leg dedup output that also carries an exact marker line could still
note falsely — vanishingly narrow, consequence is one wasted read.

## Files changed
- `server/PtkMcpServer/Tools/JobTool.cs` — anchored regex + shortened-text
  gate.
- `server/PtkMcpServer.Tests/RawUsageTests.cs` — new false-positive guard:
  a job printing the exact marker line gets NO note on an under-limit
  poll.

## Guard proof
- `RawUsageTests::Job_printed_marker_text_gets_no_false_recovery_note` —
  red leg: against the sd3-2 substring check, FAILS (false note appended);
  with the fix, PASSES, and the genuine-elision guard still passes.
